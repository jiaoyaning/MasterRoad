# 单一职责原则
>就一个类而言，应该仅有一个引起它变化的原因

# 开闭原则
> 定义的类、模块、函数等应该是可以扩展的，但是不可修改

# 里氏替换原则
> 所有引用父类的地方必须能透明地使用其子对象

# 依赖倒置原则
> 高层模块不应该依赖低层模块，两者都应该依赖抽象。抽象不应该依赖细节，细节应该依赖抽象

# 接口隔离原则
> 一个类对另一个类的依赖应该建立在最小的接口上

# 迪米特法则
> 一个软件实体应当尽可能少地与其他实体发生相互作用
